# GlobalBird 云服务器操作回顾

本文用于回顾本次对话中与云服务器相关的操作流程、技术决策、仓库内已形成的部署骨架，以及当前仍未验证或未完成的事项。

文档定位是“回顾抓手”和“后续接手入口”，不是重新撰写一份部署设计方案。正文尽量只记录本次对话中已经明确讨论、确认或落到仓库的内容，不额外扩展未出现的新设定。

本文中的事实来源分为四类：

| 状态 | 含义 |
|---|---|
| 已确认 | 本次对话中已经明确达成一致的技术决策 |
| 已落仓库 | 已经在当前仓库中新增或修改的文件与代码 |
| 服务器侧反馈 | 用户转述的云服务器 AI 检查结果，未在本地对话环境直接复核 |

## 2. 本次对话确定的核心部署方案

### 已确认

- 前端继续留在本机开发，不随当前阶段一起迁到云端。
- 后端部署到中国大陆单台阿里云 ECS。
- 家里电脑负责下载、整理和单向同步数据包到 ECS。
- ECS 负责导入、查询和对外服务，不把家里电脑作为运行时回源。
- 当前服务拓扑按容器方式组织为：
  - `nginx`
  - `api`
  - `worker`
  - `postgis/postgis`
  - `redis`

### 已确认的部署思路

```text
本机浏览器
  -> 本机 Vite Dev Server
  -> /api 代理到 ECS 公网入口
  -> ECS Nginx
  -> ECS 内部 API
  -> ECS 内部 PostGIS / Redis

家里电脑
  -> 下载或整理静态数据包
  -> 通过 SSH / rsync / sftp / Tailscale SSH 同步到 ECS
  -> ECS 本地目录
  -> worker 导入数据库
```

### 记录说明

- 对话早期曾讨论过香港/新加坡 ECS 的可能性，后续已明确切换为“国内 ECS + 家里电脑同步”的方案。
- 对话中曾有多个示例端口，最终仓库内形成的部署骨架里，API 容器端口已经固定为 `8000`，由 `nginx` 反向代理到 `api:8000`。

## 3. 云服务器侧操作/决策时间线

### 已确认的时间线

1. 最初讨论是否将后端放到阿里云服务器，并比较了香港/新加坡 ECS 与国内 ECS 的方向。
2. 因香港/新加坡 ECS 成本偏高，后续决策切换为国内 ECS。
3. 为避免国内 ECS 强依赖外网采集，进一步决定由家里电脑承担数据获取职责，ECS 只负责导入和服务。
4. 明确不将家里电脑作为运行时源站，只允许“单向同步”到 ECS。
5. 明确所有核心服务按容器方式运行，不在宿主机直接安装宿主机版 `nginx/postgres/redis` 服务。
6. 明确公网只保留统一入口，不把 `api/postgis/redis/worker` 直接暴露到公网。
7. 明确代码目录与数据目录分离：
   - 代码目录：`/opt/globalbird/app`
   - 数据目录：`/opt/globalbird/data/*`
8. 明确 worker 第一阶段的目标不是在线抓取，而是优先支持本地发布包导入。
9. 在仓库内补齐了当前阶段的 ECS 部署骨架、Nginx 配置、环境变量样例和数据库初始化 SQL。

### 当前已知的实际服务器项目根目录

- 对话后续已补充确认：ECS 上当前实际使用的仓库根目录是 `/opt/globalbird/app/GlobalBird`
- `/opt/globalbird/app` 只是父目录，不应再把它误认为仓库根目录
- 因此，后续所有服务器侧操作如果涉及项目代码，应优先以 `/opt/globalbird/app/GlobalBird` 为准

## 4. 安全组与网络策略

### 已确认

对话中确认的目标安全策略如下：

| 项目 | 目标状态 |
|---|---|
| 公网入口 | 只保留 `nginx` |
| 对外端口 | 当前阶段以 `80` 为主 |
| `3000/5432/6379` | 不开放公网 |
| `443` | 暂不开，等未来域名和证书接入后再考虑 |
| `22` | 是否开放取决于连接方式与本机网络条件 |

### 已确认的补充背景

- 对话中曾建议：如果用户有固定公网 IP，则 `22` 和 `80` 可以按最小权限放行。
- 后续用户明确表示本机没有固定公网 IP，因此对话建议改为优先通过阿里云 Workbench 管理 ECS，不依赖固定 `/32` 白名单。
- 基于这一变化，对话中形成的倾向是：
  - 优先依赖 `Workbench`
  - `22` 不再作为当前阶段的首选前提
  - `80` 仍作为公网入口

### 说明

本节记录的是本次对话中的建议目标状态，不代表当前 ECS 控制台中的安全组规则已经逐项核验。

## 5. 容器与挂载规范

### 已确认

本次对话对 5 个容器的挂载职责已经做了明确划分：

| 服务 | 挂载策略 | 说明 |
|---|---|---|
| `nginx` | 挂配置文件，不挂数据目录 | 当前重点是反向代理配置，未来域名/HTTPS 也复用该入口 |
| `postgis/postgis` | 必须挂宿主机数据目录 | 避免容器重建导致数据库丢失 |
| `redis` | 建议挂宿主机数据目录 | 便于持久化 AOF/缓存状态 |
| `api` | 当前无状态，不要求宿主机数据目录 | 状态应进入 Postgres/Redis，而不是写本地 |
| `worker` | 必须挂数据交换目录 | 读取同步数据包并处理归档 |

### 已确认的目录规范

#### 代码目录

```text
/opt/globalbird/app
```

#### 数据目录

```text
/opt/globalbird/data/postgres
/opt/globalbird/data/redis
/opt/globalbird/data/inbox
/opt/globalbird/data/archive
/opt/globalbird/data/releases
```

### 已确认的核心原则

- 代码目录与数据目录分离。
- 不把数据库或缓存状态写在容器临时层。
- 不让 `api` 与 `worker` 承担不必要的宿主机写入职责。
- 不让运行时请求直接穿透回家里电脑读取数据。
