## 2. 当前项目的正确认识

GlobalBird 当前还处在 `V1 主链路搭建` 阶段，重点不是做完整产品，而是先把下面这条链路打通：

1. 前端主视图
2. cell 聚合展示
3. cell 明细
4. 物种详情
5. 图片媒体
6. 合规与引用信息

当前不属于 `V1` 优先级的内容：

- AI 对话
- 音频处理
- 视频媒体
- GraphQL
- 移动端高性能适配

仓库结构已经固定为 monorepo：

- `apps/web`：Vue + Vite 前端
- `apps/api`：Fastify API
- `apps/worker`：ETL / 后台任务
- `packages/contracts`：前后端共享 DTO 与错误定义
- `docs`：长期文档和运维记录
- `deploy/ecs`：ECS 单机部署骨架

## 3. 本次对话已经确认的关键结论

### 3.1 开发优先级

下一阶段重点不是继续加前端视觉层，而是转到后端主链路开发，顺序已经明确：

1. 先补 `worker`
2. 再让 `api` 从 mock 切到真实数据库
3. 真实 API 稳定后，再推进正式 Three.js 场景

### 3.2 部署路线

已经明确采用这条路线，不再回到别的方向：

- 前端继续在本机开发
- 后端部署到中国大陆单台阿里云 ECS
- 家里电脑负责下载、整理、同步数据包到 ECS
- ECS 负责导入、查询和对外服务
- ECS 不把家里电脑作为运行时回源

### 3.3 容器与公网边界

已经确认当前阶段的服务形态是：

- `nginx`
- `postgis/postgis`
- `redis`
- `api`
- `worker`

原则已经定下：

- 只让 `nginx` 承担公网入口
- `5432 / 6379 / 8000` 不对公网开放
- 宿主机不直接安装宿主机版 `postgres / redis / nginx` 作为主服务

### 3.4 数据同步方式

已经确认的数据入口不是“ECS 直接稳定抓外网”，而是：

1. 家里电脑下载或整理数据包
2. 同步到 ECS 本地目录
3. `worker` 从 ECS 本地目录导入
4. API 从 ECS 本地数据库读数据

这一点已经明确，不再采用“家里电脑做线上源站”或“ECS 运行时穿透回家里电脑读取资源”的方案。

## 7. 当前最容易混淆的点

这几条需要单独记住，否则新对话很容易误判现状。

### 7.1 “服务器上容器跑起来了” 不等于 “业务已经打通”

当前最有可能已经成立的是：

- `nginx / postgis / redis` 在跑
- 数据库 schema 已初始化

但当前还不能据此认为：

- API 已经接入真实数据库
- Worker 已经具备导入能力
- 真数据链路已经打通

### 7.2 “仓库有 ECS 骨架” 不等于 “服务器已经完整跑仓库这套骨架”

仓库里的 `deploy/ecs/*` 和 `apps/*/Dockerfile` 已经存在，但服务器上是否完全切到这一套，需要继续核对和落地。

### 7.3 当前 API 的主要问题不是“端口”，而是“数据源仍是 mock”

就算 API 容器稳定跑起来，只要它还是用 [api-v1.ts](/Users/Xiang/PersonalProjects/GlobalBird/apps/api/src/routes/api-v1.ts) 里的 mock repository，它也只是“云上的 mock API”，还不是真实后端。

## 8. 接下来要做的内容

下面是建议直接按顺序执行的下一阶段任务。

### 8.1 先确认 ECS 上运行的是仓库版 API

目标：

- 让 ECS 跑 [apps/api/Dockerfile](/Users/Xiang/PersonalProjects/GlobalBird/apps/api/Dockerfile) 构建出来的 `api`
- 让 `nginx` 正确代理 `/api/` 到 `api:8000`
- 让 `/api/v1/health` 可访问

为什么先做这一步：

- 不先确认 API 的容器来源，后面所有联调都会混在“占位容器”和“仓库真实代码”之间

### 8.2 旋转数据库密码，并同步修改 ECS `.env`

目标：

- 把当前弱口令替换掉
- 让 `postgis`、`api`、后续 `worker` 都使用新的凭证

为什么现在要做：

- 数据库已经初始化完成，这时改密码成本最低
- 再往后接 API 和 Worker，会让改动面更大

### 8.3 实现 Worker 的“本地发布包导入”第一版

这是下一阶段真正的主任务。

目标不是先做 GBIF 在线下载，而是先支持：

1. 家里电脑准备一个最小发布包
2. 上传到 `/opt/globalbird/data/inbox`
3. `worker` 识别这个发布包
4. 解析并导入到 `gbs.*` 表

为什么这是当前最合理的下一步：

- 这条链路和当前选定的“国内 ECS + 家里电脑同步”架构完全一致
- 它能最早让数据库里出现真实数据
- 没有真实导入链路，API 接库也只能空转

### 8.4 把 API 从 mock repository 切到 PostgreSQL

推荐先改接口顺序：

1. `/api/v1/cells`
2. `/api/v1/cells/:h3/occurrences`
3. `/api/v1/species/:speciesKey`
4. `/api/v1/species/:speciesKey/media`
5. `/api/v1/compliance/downloads/:downloadKey`

为什么这个顺序合理：

- `cells` 是前端主视图的第一入口
- `occurrences` 和 `species` 才能支撑详情流
- 合规下载信息可以跟随真实导入链路逐步补齐

### 8.5 最后再把本机前端代理切到 ECS

等 API 真正稳定后，再通过 [vite.config.ts](/Users/Xiang/PersonalProjects/GlobalBird/apps/web/vite.config.ts) 的 `VITE_API_PROXY_TARGET` 把本机前端联调到 ECS。

这样做的好处是：

- 前端仍保留本机 HMR 和快速开发体验
- 后端走云端真实数据库
- 问题边界清晰，不会把前端问题和部署问题混在一起
