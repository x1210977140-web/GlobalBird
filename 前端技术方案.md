# 万羽拾音 Global Bird Symphony 前端技术方案

## 执行摘要

我将本前端技术方案定位为“数据密集型 3D 交互可视化 Web 应用”的工程落地蓝图：前端不追求“把全量点位直接推到浏览器”，而是严格遵循“按视野、按缩放级别（LOD）先聚合再渲染”的原则，通过后端的 H3 聚合接口按需获取当前视野内的 cell 统计，再在 WebGL 中以批处理（Instancing + 纹理图集）渲染，避免 draw call 和贴图切换爆炸，从而在百万级 occurrence 与十万级媒体资产规模下仍可保持顺滑交互。

本方案将前端架构拆为四层：UI 交互层（DOM）、3D 渲染层（Canvas/WebGL）、数据访问层（REST Client + 缓存/取消/重试）、状态与业务编排层（统一状态驱动 UI 与 3D）。后端契约以 `/api/v1/cells`（视野聚合）为入口，配合 `/api/v1/cells/{h3}/occurrences`（明细分页）、`/api/v1/species/{speciesKey}`（物种详情）、`/api/v1/species/{speciesKey}/media`（媒体列表）、`/api/v1/media/{mediaId}/signed-url`（原图签名访问）、`/api/v1/compliance/downloads/{downloadKey}`（合规引用）闭环。fileciteturn8file0

合规是前端“必须实现”的能力：在媒体与明细展示中，我会强制呈现 license、rightsHolder/creator、publisher，以及必要时的 DOI/引用信息；同时通过“缩略图走 CDN、原图走签名 URL”的策略降低带宽与加载失败风险，并对签名 URL 做短期缓存与过期重取。fileciteturn4file0

本文按“V1 主链路优先”收敛范围：前端正式联调只面向 `/api/v1` 契约，并只覆盖 3D 地球、聚合查询、明细分页、物种详情、图片媒体与合规展示；AI 与音频仅保留 feature flag 与占位目录，不进入 V1 开发与验收。fileciteturn6file0turn6file1

## 项目背景与目标

### 项目背景

从整体文件与后端技术方案可以归纳出：该项目以鸟类为主的生物多样性数据应用为主题，数据来源以 occurrence（观测/标本记录）与关联多媒体为核心；产品形态为 Web 端的 3D 交互可视化（地球/全景）应用，强调“海量点位 + 实时交互 + 浏览器性能”的同时成立。fileciteturn7file0

后端方案明确给出一条关键工程原则：避免前端直接连接第三方资源、避免把公共检索 API 当作实时数据源使用；而是通过自有 ETL 与自有 REST API 对前端提供稳定契约，并通过 PostGIS + H3 聚合降低前端渲染成本。fileciteturn8file0

### 项目目标

在不引入额外未指定需求的前提下，我将前端目标定义为：

- 交互目标：可旋转、缩放、对焦的 3D 地球视图；支持 hover/click 拾取与信息面板展示；在视野变化时能稳定刷新数据并保持交互连贯。
- 数据目标：前端只消费“当前视野 + 当前 LOD”的数据；默认使用聚合数据渲染，按需加载分页明细；不得出现“全量点位一次性加载”的实现路径。fileciteturn7file0turn8file0
- 合规目标：合规字段永不丢失，媒体与明细必须展示 license 与归属信息，并在合适位置展示引用/DOI。fileciteturn8file0
- 工程目标：可测试、可部署、可回滚、可观测；关键链路出现错误可通过 requestId 定位。fileciteturn7file0turn8file0

## 关键约束与假设

### 关键约束

下表为我从两份文件中提取的“对前端落地有直接影响”的硬约束与工程约束。fileciteturn4file0turn5file6

| 维度 | 约束内容 | 对前端的直接影响 |
|---|---|---|
| 数据规模 | 百万级 occurrence、十万级媒体资产 | 必须 LOD + 聚合优先；禁止一次性拉全量；渲染必须批处理 |
| 数据获取原则 | “聚合优先、明细按需”，视野 bbox + h3Res 返回 cell 聚合 | 前端所有地图/视野请求都要带 bbox + h3Res；并实现请求取消/合并 |
| 缓存窗口 | `/cells` 可接受 30s~5min 近实时；服务端返回 `cache.ttlSeconds` | 前端需要尊重 TTL；视野快速变化要取消旧请求，避免触发限流 |
| 合规字段 | 必须保留并展示 license、rightsHolder/creator、publisher；必要时 DOI/引用 | UI 必须设计“归属/许可区块”；缺失字段要降级展示并记录数据质量问题 |
| 媒体策略 | 缩略图走 CDN；原图默认签名 URL；签名 URL 有时效；前端需过期重取与失败重试 | 需要 signed-url 获取器（鉴权、短期缓存、过期检测、重试/降级） |
| 鉴权与安全 | signed-url 至少需要 API key 或 session；CORS 只允许受信任域名；聚合与签名接口要限流 | 前端必须实现登录/令牌刷新或 API key 承载策略；并实现请求合并、节流、退避 |
| H3 分辨率 | 建议前端传入 h3Res，后端校验并纠正到允许集合（示例 5~9） | 前端需要稳定的“缩放→h3Res”映射与滞回；并限制参数范围 |
| 环境策略 | 至少 dev/staging/prod 三套环境隔离 | 前端需多环境配置与构建产物区分；并支持 feature flag |

## 前端总体架构路线图

### 分层架构

我将前端拆为“可独立演进、可独立测试、可独立替换”的四层：fileciteturn7file0

1. UI 交互层（DOM）：筛选、搜索、卡片、抽屉、设置、提示与空态等。
2. 3D 渲染层（Canvas/WebGL）：地球场景、marker 批渲染、LOD 表现、拾取与交互反馈。
3. 数据访问层：REST 客户端、请求取消/合并、缓存、重试、错误码映射、鉴权注入。
4. 状态与业务编排层：统一状态驱动 UI 与 3D（如 selected cell/species、camera pose、filters、feature flags）。

这四层的核心收益是：UI 与 3D 解耦（降低互相拖慢）、数据访问策略集中（避免各处重复造轮子）、状态一致性可检验（可测试、可回放）。

### 目录结构与模块边界

我推荐使用“按领域模块 + 横切能力 core”的结构（示例）：

```text
src/
  app/                 # 应用入口、路由、全局装配
  core/                # 横切能力：env、logger、error、http、cache、feature-flag
  modules/
    scene/             # 3D 场景：渲染循环、资源管理、拾取、实例化渲染
    map-lod/           # 缩放→h3Res 映射、滞回、bbox 计算、视野节流
    query/             # 领域查询：cells、occurrences、species、media、compliance
    media/             # 媒体展示：缩略图、预览、原图签名、归属信息
    audio/             # 音频播放（可选）：WebAudio 封装、预加载、错误与兼容处理
    ai/                # AI 对话（可选）：占位 UI、协议定义、feature flag
  pages/               # 路由页：Globe、Species、About/Compliance、Debug（可选）
  components/          # 通用 UI 组件（尽量无业务）
  assets/              # 静态资源（小图标、基础纹理）；大纹理建议走 CDN
  styles/              # 全局样式、设计 token
  types/               # DTO/契约类型（可与后端共享：建议单仓或包发布）
```

### 部署拓扑与关键流量路径

为了满足“静态资源高命中 + API 可控 + 媒体分发有成本边界”，我采用“三域/三层”思路（均可通过配置替换）：fileciteturn5file6turn4file0

- 应用静态资源：CDN/静态站点（`app` 域）
- 后端 API：业务 API 域（`api` 域，含 `/api/v1/...`）
- 媒体分发：媒体 CDN 域（`media` 域，缩略图/预览图）；原图走 signed-url（可能直接回源对象存储或经 CDN）

下面给出“Mermaid 流程图语法的纯文本表达”（不依赖渲染），并附等效文字说明。

```text
flowchart TB
  U[用户浏览器] --> A[CDN: 前端静态资源]
  U -->|REST| B[API: /api/v1/*]
  U -->|缩略图/预览图| C[媒体CDN]
  B -->|返回 cell 聚合| U
  B -->|返回物种/媒体列表| U
  B -->|签名URL| U
  U -->|原图 signed-url| D[对象存储/回源链路]
```

等效文字说明：

1. 首屏：浏览器从 CDN 拉取 index.html、JS/CSS 与必要静态资源。
2. 数据：前端 JS 以 REST 调用后端 `/api/v1/cells` 等接口获取聚合与详情数据。
3. 媒体：列表页与卡片优先加载 CDN 缩略图（长缓存）；当用户请求原图时，先向后端请求签名 URL，再使用签名 URL 拉取原图资源。
4. 安全：签名 URL 接口必须鉴权并限流；前端必须合并请求并在视野变化时取消旧请求。

### CI/CD 与版本管理策略

CI/CD 我建议对“前端静态站点”采用轻量流水线：fileciteturn7file0turn5file6

- PR：lint → typecheck → unit test → build（产物不发布）
- main：build → 产出静态资源 → 上传对象存储/静态托管 → CDN 刷新（或版本化路径）→ 健康检查（可选）

版本管理策略：

- 代码版本：trunk-based 或简化 Git Flow 均可；但我更推荐“短生命周期分支 + 主干集成”，以减少 3D/性能类改动的长分支漂移风险。
- 发布版本：前端产物以“构建号 + Git commit hash”标识；静态资源建议走内容哈希文件名（Vite 默认行为），以便 CDN 长缓存与回滚（回滚只需切换 index.html 指向的构建版本）。

### 多环境配置与 feature flag

- 环境（dev/staging/prod）：决定 API Base、媒体域名、日志级别、是否打开调试面板。
- feature flag：决定是否启用 AI、音频、原图入口、调试覆盖层等。

### 数据流与状态模型

#### 领域状态拆分

我推荐把状态拆为以下 store（示例）：

- `sceneStore`：camera pose、渲染开关、帧率采样、拾取状态（hoverId、selectedId）。
- `filterStore`：yearFrom/yearTo、mediaType、datasetKey 等筛选条件。
- `lodStore`：当前 h3Res、当前 bbox、滞回阈值、请求节流参数。
- `queryStore`：cells 聚合缓存、occurrence 分页缓存、in-flight 请求表（用于去重/取消）。
- `uiStore`：抽屉打开状态、当前 tab、空态/错误态。
- `authStore`（若需要）：登录态、token/session 刷新状态、权限集（例如是否允许取原图）。

#### 视野聚合数据结构建议

后端 `/cells` 返回结构偏业务；为了减少 3D 更新成本，我会在前端把它转换为“面向 GPU 的结构化数组”。fileciteturn6file1turn7file0

```text
CellAggGPUBuffer
  instanceCount: N
  positions: Float32Array(3N)        # center lon/lat -> xyz(转换后)
  sizes: Float32Array(N)             # 由 occurrenceCount 映射
  colors: Float32Array(3N)           # 由 speciesCount 或热度映射
  ids: Uint32Array(N)                # 索引/映射到 h3
  meta: Map<id, {h3, occurrenceCount, speciesCount, topSpecies...}>
```

这样做的好处是：当 `/cells` 刷新时，我只需要批量更新 TypedArray 并一次性 `needsUpdate`，避免频繁创建/销毁对象导致 GC 抖动。

### 关键接口定义与前端契合点

以下接口为后端技术方案已经明确的“前端应按此联调”的契约。我给出请求参数、返回示例与前端处理要点（示例 JSON 均为示意）。fileciteturn6file1turn4file0

#### 获取视野聚合 cell

- 方法：GET  
- 路径：`/api/v1/cells`  
- 参数：
  - `h3Res`：整数（建议前端限制在 5~9 范围内）
  - `bbox`：`minLon,minLat,maxLon,maxLat`
  - 过滤参数：`mediaType`、`yearFrom`、`yearTo`（后端示例已给出）

前端关键处理：

- 必须做参数本地校验；非法应在前端回退默认值或提示（对应 400 INVALID_ARGUMENT）。
- 视野变化触发新请求时，要取消旧请求；同一 bboxHash + filterHash 在 TTL 内允许复用缓存。
- 以 `cache.ttlSeconds` 作为“该响应可复用时间”的上限提示，不要超过服务端声明。

#### 获取 cell 明细分页

- 方法：GET  
- 路径：`/api/v1/cells/{h3}/occurrences`  
- 参数：
  - `limit`：建议 50~200（结合 UI 列表与性能）
  - `cursor`：后端给出的游标字符串

前端关键处理：

- 明细列表应做虚拟滚动或分页 UI；避免一次性渲染大量 DOM。
- 在用户持续缩放/移动时，明细请求应取消或延迟；避免与 `/cells` 争抢带宽与主线程。
- 对 404 NOT_FOUND 展示空态；对 503 可降级显示缓存或提示只读。

#### 获取物种详情与媒体列表

- 方法：GET  
- 路径：`/api/v1/species/{speciesKey}`、`/api/v1/species/{speciesKey}/media`  
- 参数（media）：
  - `type`：示例为 StillImage
  - `limit`：如 20~50

前端关键处理：

- `/species` 与 `/media` 属于相对稳定数据，可做较长 TTL 的本地缓存（内存或持久化）。
- 媒体 UI 必须展示 attribution（license/rightsHolder/creator/publisher/source.identifier 等）。

#### 获取原图签名 URL

- 方法：POST  
- 路径：`/api/v1/media/{mediaId}/signed-url`  
- 鉴权：必须（至少 API key 或 session）

前端关键处理：

- 对同一 mediaId 的 signed-url 建议做 30~120 秒短缓存；避免频繁签名计算。
- 若访问原图失败（403/过期），应触发“重取 signed-url”流程；若仍失败，降级仅显示预览图并提示。

#### 获取合规与引用信息

- 方法：GET  
- 路径：`/api/v1/compliance/downloads/{downloadKey}`

前端关键处理：

- 建议把“数据来源/合规信息”独立成页面或弹窗；在媒体页也提供入口。
- 若缺 downloadKey 或后端未提供数据，显示“未指定”并提示联系管理员。

### 统一错误处理、重试与降级策略

后端已给出错误码表；我建议前端统一实现以下策略矩阵：fileciteturn4file0turn3file5

| code | 典型场景 | 我在前端的处理策略 |
|---|---|---|
| INVALID_ARGUMENT | bbox/h3Res/过滤条件不合法 | 前置校验 + 自动回退；记录日志与埋点 |
| UNAUTHORIZED | signed-url/私有能力 | 触发登录或刷新令牌；若无登录能力则隐藏入口 |
| FORBIDDEN | 无权限看原图 | 降级：只展示缩略图/预览图；提示权限不足 |
| RATE_LIMITED | 拖动视野请求过密 | 合并请求 + 取消旧请求 + 指数退避；必要时锁定 UI 1~2 秒 |
| INTERNAL_ERROR/SERVICE_UNAVAILABLE | 服务端异常/维护 | 展示可恢复错误态 + “重试”按钮；对 `/cells` 可先用缓存数据继续渲染（标注非实时） |

### 数据与状态管理细则

#### 本地缓存策略

- `/cells`：只做内存短期缓存（与后端 30s~5min 窗口一致），key = `h3Res + bboxHash + filterHash`；不做持久化，避免过期数据误导。fileciteturn4file0turn3file9
- `/species` 与 `/media`：可做内存 + 持久化（IndexedDB，可选）；TTL 建议 6h~24h，且支持“版本化失效”（当后端数据版本升级时整体失效）。fileciteturn3file9
- signed-url：严格短缓存（30~120 秒）；过期或 403 即失效重取。fileciteturn4file0

#### 离线支持

未指定“离线必须可用”。我建议：

- 基线：仅缓存静态资源，确保弱网可打开页面。
- 可选：缓存最近访问的物种详情与媒体缩略图；明确标注为“离线缓存内容，可能不是最新”。

#### 一致性与并发控制

- `/cells` 属于高频数据：以“最新请求优先、旧请求丢弃”为原则；任何渲染更新都必须检查 request sequence（避免乱序覆盖）。
- `/occurrences` 分页：允许并发加载下一页，但必须保证游标顺序与去重。
- UI 状态更改（如 filters）：采用“悲观更新”（先更新 filters，再触发数据刷新）；不做乐观写入（当前无写接口）。

### 测试与质量保障

在整体文件的建议基础上，我把测试拆为三层，并给出覆盖率目标（可按团队规模调整）：fileciteturn5file0turn7file0

- 单元测试（目标覆盖率：核心纯函数与协议层为主，建议 70%+）：  
  - 经纬度→球面坐标、缩放→h3Res 映射（含滞回）、颜色/大小映射函数、bbox 计算、hash/key 生成、错误码映射等。
- 集成测试（目标：关键模块联动稳定）：  
  - 3D 场景初始化、数据到 GPU Buffer 更新、拾取→UI 打开、分页加载与取消。
- E2E（目标：覆盖核心用户旅程，建议 10~20 条用例起步）：  
  - 打开首页 → 拉取 cells → 可拖拽缩放 → 点击 cell → 展示明细 → 点击物种 → 展示媒体 → 查看原图（若有权限）→ 进入合规页。

代码审查规范（建议）：

- 任何新增第三方依赖都需要在 PR 描述中说明：体积影响、替代方案、升级策略与安全风险。

### 性能优化清单与度量指标

#### 性能优化清单

- 首屏：3D 引擎与大资源延迟加载；先渲染“可交互的地球骨架”，再加载 cell 数据与纹理细节。
- 渲染：Instancing + TypedArray 批更新；避免频繁 new 对象；限制透明 overdraw；纹理 atlas 减少切换。
- 交互：mousemove 拾取节流；两级拾取缩小候选集；对焦动画可取消/打断。
- 网络：视野变化取消旧请求；请求合并；对 429 做指数退避；对 `/cells` 复用缓存与 TTL。
- 媒体：缩略图/预览图优先；原图按需加载；图片解码与预加载控制并发。fileciteturn7file0turn4file0

#### 建议度量指标

- 帧率：平均 FPS、P95 帧时间（尤其是交互/拖拽时）
- CPU/GPU：主线程长任务次数（Long Task）、内存峰值与 GC 次数
- 网络：`/cells` 请求频率、取消率、P95 延迟、429 比例
- 质量：JS 错误率、资源加载失败率、签名 URL 失败率

### 部署与运维

#### 构建产物与发布

- 构建产物：静态资源（HTML/JS/CSS/字体/小图标）；大纹理与模型建议走独立 CDN bucket。
- 发布：建议版本化目录发布，例如 `/releases/{buildId}/`；`index.html` 通过短缓存或原子切换指向目标版本，实现回滚。
- 灰度：若需要灰度，可按用户/地域/百分比在 CDN 或网关层分流（未指定，作为可选）。

#### 回滚策略

- 前端回滚：切换 `index.html` 指向旧 buildId（静态资源带 hash，命中率高）。
- 联动回滚：若 API 契约有破坏性变更，前端必须做“兼容层”（例如容忍字段新增/缺失），并通过 feature flag 控制启用新能力。

### 风险识别与缓解措施

| 风险 | 影响 | 触发信号 | 缓解措施 |
|---|---|---|---|

| 数据接口契约变动 | 联调反复、线上错误 | 字段变更、路径变更、分页策略变更 | DTO 版本化；前端容错解析；feature flag 切换 |
| 请求风暴触发限流 | 429 增多、体验差 | 拖动视野时 429 比例上升 | Abort + 去重 + 节流；缓存复用；UI 限制刷新频率 |
| 媒体签名与权限不清 | 原图不可用、合规风险 | 403/401 高频；用户困惑 | 明确权限模型；对原图入口做显式提示与降级 |
| 合规遗漏 | 法务/声誉风险 | 归属/许可字段缺失或未展示 | UI 强制展示区块；字段缺失标记为数据质量问题；合规页集中说明 |
| 范围膨胀（AI/音频提前进入） | V1 延期、联调失控 | 新增页面或接口进入当前迭代 | 把 AI/音频锁定为 V1 后事项；需要单独 RFC 与排期后再接入 |
